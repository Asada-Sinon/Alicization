# 水系统修复:三候选对比与选型判决

`docs/water_system.md` 把幼体渴死瓶颈(81.9% 死亡)确立为总闸门,并证明"调水经济"
是第一个能真正松动它的杠杆。随后三个并行方案各自实现并 6 配对种子实测了一个候选修
复。**这份文档把三者放在一起,对着用户预先给定的判据做选型,并记录被选中方案作为
main 默认值落地的决定。** 标记约定沿用 `docs/biology.md`。

三份来源文档,读它们看机制与完整逐种子数据:

- `docs/water_fix_buffer.md` —— 失水耐受缓冲(`water_deficit_buffer`,改 `cull` 的
  `parched` 判定,水可短暂为负)
- `docs/water_fix_provisioning.md` —— 解耦新生供水(`water_lactation_floor_frac`,
  水交接按 `max(invest_frac, floor)`,能量交接不变)
- `docs/water_fix_retune.md` —— 水阀门 + 捕食者补偿旋钮(纯 `--set`:耗水减半 +
  `carn_cost` 0.10→0.15)

---

## 0. 一句话结论

**选 retune(`base_water_cost=0.01`、`move_water_cost=0.025`、`carn_cost=0.15`)作为
main 新默认值。** 它是三者中**唯一**同时满足用户预先登记的三条判据的方案:渴死大降
(0.829→0.540)、捕食者离水(`carn_water_dist` 11.4→25.3)、`carn_frac` 不升反稳
(0.241→0.216,统计不可区分于基线)。另两个机制作为默认关闭的 flag 已合并进 main
(研究留痕),但都不作为激活的默认值:buffer 把 `carn_frac` 顶到 41%(判据③直接违
反);provisioning 安全但**碰不到用户第一诉求**——它让捕食者离水更近而非更远。

被选方案有一个**必须正视、但不在用户判据内**的代价:种群 +79%。它与用户口头提过的
"满屏食草动物"观感相反,但 `docs/rebalance.md` 已独立判决"降种群"是一条正交的、
单独走不通的轴,不在水修复的范围内。retune 是三行 config 改动 + golden 重新定标,一次
`git revert` 即可回退,不作废任何演化种群。

## 1. 用户预先登记的判据

用户在授权本轮自主执行时给的理想标的(逐字沿用记忆里的登记):**渴死占比大降 +
捕食者离水 + `carnivore_frac` 不膨胀过 ~23%**。这三条是选型的硬判据。种群规模不在其
中——这是有意的,因为 `docs/rebalance.md` 已经把"降种群"划为单独的、方向相反的问
题。下面按这三条逐一核对。

## 2. 三候选对比(全部 6 配对种子,20000 步,种子 0–5)

| 指标 | 基线 | buffer(deficit=1.0) | provisioning(floor=0.7) | **retune** |
| --- | --- | --- | --- | --- |
| `death_thirst_frac` | 0.823–0.829 | **0.511**(大降) | 0.835(**几乎不动**) | **0.540**(大降) |
| `death_thirst_age` | ~48–50 | 204(+震荡) | 56.3(+16%) | (渴死更晚,未单列) |
| `carn_water_dist`(离水) | 11.4 | (升,与水松弛同向) | **9.3(更近!错向)** | **25.3(离水,+122%)** |
| `carnivore_frac` | 0.236–0.241 | **0.412(顶到 41%)** | 0.197(降) | **0.216(稳,p=0.094)** |
| `population` | ~1140–1166 | +52–96% | 1154(**持平**) | **2032(+79%)** |

(buffer 与 provisioning 的空间量口径略有差异,但方向清楚;精确逐种子数字见各自来源
文档。)

## 3. 判据逐条核对

**① 渴死占比大降。** buffer ✓(0.51)、retune ✓(0.54)、provisioning ✗——
provisioning 的渴死**占比**几乎不动(0.823→0.835)。它确实让幼体晚死(`thAge`
+16%,p=0.031),但那是竞争性风险的重分布:被救下的幼体没死于渴,改死于别的原因,
渴死在总死亡里的**份额**没降。这是一个诚实但致命的区分——用户判据要的是"占比大
降",不是"死得晚一点"。

**② 捕食者离水。** **只有 retune 满足,且是决定性的。** `carn_water_dist`
11.4→25.3(+122%,6/6 同向,p=0.031)。这直接回应用户反复提的第一诉求"捕食者依旧
住在河里"。provisioning **反向**——它让捕食者贴水更近(11.4→9.3),因为解耦供水
放大了幼体存活、猎物基础在河道热点上更密,捕食者更没有理由离开。buffer 与水松弛同
向(离水),但见判据③它被否。

**③ `carn_frac` 不膨胀过 ~23%。** retune ✓(0.216,统计不可区分于基线)、
provisioning ✓(0.197,反而降)、**buffer ✗✗——顶到 0.412**。buffer 走的是
`water_system.md` 反复警告的那条链:松弛整体水约束 → 更多幼体活 → 猎物基础暴涨 →
养得起更多捕食者。它救幼体的力度最大,但把 `carnivore_riparian.md` 已指出的"捕食者
偏多一个数量级"问题又加重了近一倍。判据③一票否决 buffer。

**综合:** buffer 挂在③,provisioning 挂在①和②,**retune 三条全过**。

## 4. 为什么不是"精确打击"方案更优

`water_fix_retune.md` §4.4 自己把 retune 定位成"fallback",理由是:若"精确打击"
(buffer / provisioning,设计上不产生 `carn_frac` 副作用)有效,它们比"开阀门再找补
偿点"更简单。**实测把这个先验预期推翻了:**

- buffer 这个"精确打击"**并不精确**——它的确不需要第二个旋钮,但它自己就把
  `carn_frac` 顶到 41%,副作用比它要解决的问题更严重。"精确打击不产生副作用"是设
  计假设,实测不成立。
- provisioning 是真正无 `carn_frac` 副作用的那个(甚至反向压低),但代价是它对用户
  第一诉求(捕食者离水)完全无效、甚至反向。它"安全"是因为它几乎没动那条空间链。

所以"简单且无副作用"的理想候选并不存在:能真正松动空间病理的杠杆,都要付出某种代
价。retune 付的代价(种群膨胀)恰好落在用户判据之外,而 buffer 付的代价(捕食者占比)
恰好落在判据之内。这是 retune 胜出的根本原因,不是它"更干净"。

## 5. 被选方案的已知代价与边界

1. **种群 +79%,与"满屏食草动物"观感相反。** 但 min_pop 6 种子全部 ≥1771、离
   `n_max=16384` 硬上限充裕,不是崩溃或失控。降种群是 `docs/rebalance.md` 判定的**正
   交、单独走不通**的轴,不在水修复范围。若日后要治理观感拥挤,那是另一条线的工作,
   retune 不foreclose它。
2. **窄工作点,不可casually移植。** `carn_cost` 每 +0.05 断崖:0.15→占比 0.24(健
   康),0.20→0.10(逼近共存下限),0.25+ 开始团灭。**采纳 0.15 之后,任何对水阀门
   幅度、地形、或其他 ecology 参数的改动都必须重新标定 `carn_cost`,不能假设 0.15 可
   移植**(`water_fix_retune.md` §4.2/5.1)。这条已写进 `config.py` 的 `carn_cost`
   注释。
3. **只跨 founder 种子,未跨地形种子。** 所有空间数值严格只对这张河流地形成立(全
   项目同一限制,`terrain.build` 不吃 RNG)。
4. **golden band 会移动,这是预期的、正确的。** 改默认值本就应触发 golden 失败并重新
   定标(`--bless`),这与两个 flag 默认关闭"不动 golden"是相反且正确的:flag 是可
   选机制,retune 是新基线。

## 6. 落地记录

- 三个机制的 flag(`water_deficit_buffer`、`water_lactation_floor_frac`)已合并进
  main,默认 0,作为可选消融保留;两者在 `reproduction.reproduce` 的水交接处组合共存
  (floor 定分数,buffer 的 `max(·,0)` 守它乘的底)。
- retune 作为**激活默认值**写入 `config.py`:`base_water_cost` 0.02→0.01、
  `move_water_cost` 0.05→0.025、`carn_cost` 0.10→0.15。golden 用 `--bless` 重新定标。
- 这是一次自主决定(用户睡眠中授权,判据由用户预先登记,retune 唯一满足)。可一次
  `git revert` 回退。
