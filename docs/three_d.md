# 3D 化可行性调研:渲染、模拟、生态三本账

## 0. 这份文档为什么存在,以及怎么读

几周前做过一次几乎一样的调研。用户当面问、agent 口头汇报结论、没人把它写下来,
结论随对话上下文一起在压缩里蒸发,几周后同一个问题被原样重新提出。这份文档是
`CLAUDE.md`"调研结论必须落进 `docs/`,否则视为没做过"这条规则的直接原因和第一个
践行对象——所以本文档不满足于复述记得住的那部分,而是把每一条能核实的都重新核实
一遍,记不清或核实不了的诚实地标成记不清。

**标注约定**(沿用 `docs/biology.md` 的写法):

- `[现实]`——已发表的事实,给出作者、年份、期刊/会议
- `[本世界实测]`——本次调研在这台机器、这份代码上跑出来的数字
- `[对应]`——这条结论落在代码的哪个文件、哪个函数
- `[推导]`——本次调研做的解析/量级推导,不是测出来的,也不是文献里抄来的
- `[提案,非结论]`——一个可以做但没有做、没有测过的方案
- `[待核实]`——试图核实但没有查到可靠来源,如实说不知道,不装作知道

**用户的原始问题**："可以开一个 subagent 去调研怎么转三维场景,尝试给出一个可行
方案,单卡渲染够不够?不够的话可以暂时不升级三维加 z 轴"。硬件是单张 RTX 4090
(24 GB)。这份文档要回答的正是这个问题,但会先把"转三维"拆成三条含义完全不同的
路线——这是丢失的那份旧报告做对的地方,也是本次调研确认无误、继续沿用的框架。

**这次调研做了什么、没做什么**:

- 读了 `underworld/` 全部核心模块(`spatial.py`、`sensors.py`、`dynamics.py`、
  `terrain.py`、`state.py`、`memory.py`、`brain.py`、`step.py`、`config.py`、
  `reproduction.py`、`genome.py`)和 `server/protocol.py`、`web/render.js`、
  `web/main.js`。
- 在这台 4090 上跑了一系列 `scripts/run_headless.py` 与独立 JAX 脚本的**真实
  benchmark**,测 2D 基线在不同 `n_max`/`grid`/`retina_sectors`/`k_neighbors`/
  `sense_grid` 下的 steps/s 与显存峰值。
- **没有**实际把仿真核心改成 3D——那本身是一个大工程,不是一次调研的范围。凡是
  "3D 版本会怎样"的数字,都来自(a)对现有 2D 代码做维度替换后的独立原型脚本
  实测,或(b)对现有 2D 实测数字做解析外推,两者都会标清楚,不会混着写成"实测"。
- 派了两个子 agent 并行工作:一个专门写一个**脱离仓库的 JAX 原型**去实测 3D 空间
  索引(3×3×3 分箱+邻居 gather)相对现有 2D 索引的真实倍数;一个专门核实"遭遇率
  随维度如何变化"这个生态学论断有没有真实文献支撑。两者的结果都并入了下面对应的
  小节,并保留了它们各自的方法论说明。

**本次调研最大的方法论教训,直接来自这次实测过程本身**:这台 4090 在测试期间
被另外几个并行 agent 的 `run_headless.py` 扫描任务、以及一个无关的
18 GB 显存、100% SM 占用的策略服务器进程同时占用过。同一个 2D 基线配置在不同
时刻测出过 135 steps/s 和 591 steps/s——纯粹是显卡争用造成的 4.4 倍差异,
与代码或配置毫无关系。**这正是丢失的旧报告翻车的地方**(CPU 微基准冒充 GPU
端到端),本次调研的应对方式是:每个配置至少测 4 个连续 chunk、丢弃第一个
（编译后异常慢，原因不明，怀疑是 GPU 频率爬升或某种一次性初始化开销）、
在多次重复测量里给出**范围**而不是单一数字,并且优先信任显存峰值——显存几乎
不受计算争用影响,是本文档里最硬的一类数字。

## 1. 三条路线,先拆开,再各自算账

"转三维"在对话里经常被当成一件事,但落到代码上是三件完全不同的事,touch 的文件、
是否作废种群、是否要动 wire 协议,三条路线互不相同。

### 路线 A——3D 渲染,模拟仍是 2D 平面

模拟核心(`underworld/` 整个包)**一行不改**。世界仍然是一张 `[grid, grid]` 的
平面场,agent 仍然只有 `(x, y)` 两个坐标分量、一个标量朝向。变的只是浏览器怎么
把这些数据画出来——从当前"正交俯视、地形用明暗贴图假装浮雕"变成"真正的透视相机
+ 从高度场生成的三维网格 + agent 仍然是纹理化的点精灵,但受透视影响近大远小"。

### 路线 B——离散分层(2.5D),比如"地面/树冠"两层

世界仍然是一张平面场,但增加一个**离散**的垂直状态:agent 处在"地面层"还是
"树冠层"(或更多层),层间可以移动(付出代价),不同层看到、吃到、能被捕食到的
东西不同。这不是给 `pos` 加第三个连续分量,而是给状态加一个类似 `stratum` 的
标量或整数——概念上更接近"游戏里的楼层"而不是"物理里的高度"。

### 路线 C——真 3D:连续 z 轴,几何/感知/捕食全部三维化

`pos` 变成 `[n_max, 3]`,朝向变成某种三维姿态表示,地形从一张 2D 高度场变成
某种体素场或"高度场 + 垂直剖面"的混合结构,视网膜从平面扇区变成球面分区,邻居
索引从 2D 网格分箱变成 3D 网格分箱,捕食判定的球体/球台几何都要重新推导。这是
三条路线里唯一"名副其实"的 3D。

下面每一节都按这三条路线分别给出:改哪些文件/函数、`in_dim`/`out_dim`/
`genome_size` 变不变(即是否作废现有演化种群)、wire 协议要不要跟着动、渲染
管线改多少、以及代价的量级。

## 2. 逐路线拆解

### 2.1 路线 A:3D 渲染 + 2D 模拟

#### 要改哪些文件

**只有 `web/` 下的两个文件**,`underworld/`、`server/` 一行不动:

- `web/render.js`——目前 300 行,`[对应]` 我读过的实际结构是:一个铺满屏幕的
  quad(`PLANT_VS`/`PLANT_FS`)画地形+植物背景,叠加一次 `gl.POINTS` 画全部
  agent(`POINT_VS`/`POINT_FS`)。**没有任何投影/视图矩阵**——`PLANT_VS` 直接
  把 `[-1,1]` 的 quad 坐标当裁剪空间坐标用,`POINT_VS` 把世界坐标除以
  `u_world` 再映射到 `[-1,1]`,是纯粹的正交平移缩放,没有 `mat4` 出现过一次。
  路线 A 要做的实质工作是三件:(a) 引入真正的相机(视图矩阵+透视投影矩阵),
  (b) 地形从一张纹理化的平面 quad 变成从高度场生成的三角网格,让透视相机下
  真的能看出山有多高,(c) 拾取（点击选中 agent）从"线性逆映射"变成"对高度场
  做射线相交"。
- `web/main.js`——`worldFromClient`/`canvasFromWorld`（目前在 `render.js` 里,
  是纯粹的世界坐标⇄画布像素线性映射）在有相机之后必须换成"逆视图投影矩阵
  + 与地形网格求交",点击选中逻辑(`canvas.addEventListener("click", ...)`,
  `web/main.js:133`)需要跟着换掉底层的坐标反解,上层"找最近 agent"的逻辑
  不用变。

`server/protocol.py`、`underworld/terrain.py`、`underworld/*` 全部不动。

#### `in_dim`/`out_dim`/`genome_size`:完全不变,不作废任何种群

这是路线 A 最大的优点,值得强调:大脑输入输出维度只取决于仿真怎么感知世界,
和世界怎么被画出来毫无关系。当前 `genome_size=1381`
(`in_dim=67, hidden=16, out_dim=2, trait_dim=3`,`[对应]` `underworld/config.py`
的 `in_dim`/`genome_size` 两个 `@property`,`[本世界实测]` 用
`scripts/run_headless.py` 打印验证过)在路线 A 下原样保留,**已经演化出来的
种群、`docs/trait_evolution.md` 与 `docs/experiments.md` 里记录的全部基线数字
都继续有效**——这是三条路线里唯一一条不需要"作废种群、重新演化"的。

#### wire 协议:一个字节都不用改

`server/protocol.py` 现在是 v6:68 字节头 + agent 数组(每个 20 字节:
x, y, diet, energy, id 五个 f32)+ 平面场(`plant`/`fruit` 两张 `grid×grid`
的 u8)。地形本身**已经**通过一次性的 `UNTR` 消息(`encode_terrain`,魔数
`b"UNTR"`)在连接时和 reset 后发给客户端,`[对应]` `web/main.js` 里
`magic[0]===85 && ...` 判断魔数、`parseTerrain` 解出 `height`/`forest`/`water`
三张 u8 平面,`Renderer.setTerrain` 把它们打包成一张 RGB 纹理上传。`PLANT_FS`
现在就已经在采样 `u_terrain.r`(高度平面)做"浮雕明暗"("Relief shading" 那段,
`web/render.js:76-86`)——换句话说,**客户端手里早就有完整的高度图**,路线 A
需要的全部地形数据这条线上已经在传。真正要做的不是"多发一份数据",而是"用同一份
数据多做一步几何计算"(从纹理采样值变成网格顶点位移),这是纯客户端渲染管线
改动,不涉及 wire 格式,不需要碰 `_HEADER`/`HEADER_BYTES` 里的任何一个字段。

（这一条经过主会话交叉核实:`[对应]` `server/protocol.py:63-85`
`encode_terrain`/`web/main.js` 的 `UNTR` 分支/`web/render.js:163-176`
`setTerrain`。）

#### 渲染管线要改多少:成本量级

- **地形网格三角形数**:`[本世界实测/推导]` 当前 `grid=128`(`config.py`
  默认值),把每个格子中心当顶点、`(grid-1)×(grid-1)` 个格子各切两个三角形,
  三角形数 = `2 × 127² = 32,258`。这个数字与丢失的旧报告口头转述的完全一致
  ——说明旧报告这部分是认真算过的,不是编的。`[推导]` 顶点数 `128×128=16,384`,
  如果沿用现在这套"静态几何 + 每帧只更新纹理"的模式(地形是静态的,一次
  `new_world` 内不变),顶点缓冲和索引缓冲都只需要在收到 `UNTR` 消息时上传
  **一次**,之后每帧的渲染成本只是"用已经在显存里的网格 + 已经在每帧更新的
  `plant`/`fruit` 纹理再画一次"——这是经典的"高度场纹理位移"技巧:顶点着色器
  里对每个顶点采样高度纹理、沿法线方向位移,顶点缓冲本身只需要存 UV/网格坐标,
  不需要每帧重新计算或重新上传顶点位置。索引缓冲 `32,258 × 3 = 96,774` 个索引,
  按 `grid=128` 顶点数(16,384 < 65,536)可以用 16 位索引(`191KB`);
  `grid≥256` 时顶点数达到或超过 `65,536`,必须换成 32 位索引(`OES_element_index_uint`
  扩展,WebGL1 下已经是事实标准,现代浏览器都支持),这是唯一一个和网格分辨率
  挂钩的真实限制,而不是三角形数或带宽的限制。
- **带宽**:`[本世界实测/推导]` 逐帧新增数据量为零——agent 数组仍是
  `n×20` 字节,`plant`/`fruit` 平面仍是 `grid²×2` 字节。以 `grid=128`
  (`n_cells=16384`)、活体数 ~1000 估算:`68`(头)`+1000×20`(agent)
  `+16384×2`(平面场)`≈ 20,068 + 32,768 + 68 ≈ 52.9 KB/帧`,与旧报告口头
  转述的"51.6 KB/帧"几乎完全吻合,这条数字这次也复核通过了。30 帧/秒下约
  1.6 MB/s,任何局域网或本机连接都不是瓶颈。
- **点精灵是否要变**:`gl.POINTS` 的 `gl_PointSize` 本来就是在顶点着色器里
  设的一个标量,一旦真正接入透视投影,`gl_Position` 经过透视除法之后自然
  "近大远小",现有 `POINT_VS`/`POINT_FS` 的核心逻辑(按 `diet` 混合颜色、
  按 `energy` 调大小/亮度)**基本可以原样保留**,只需要把 `gl_Position` 的
  计算从"世界坐标线性映射到裁剪空间"换成"world→view→clip 的标准矩阵乘法"。
- **相机与拾取**:需要一个最小的 4×4 矩阵库(WebGL1 无内建、项目又没有
  `node_modules`/打包步骤,不能 `npm install gl-matrix`,得自己写或内联一个
  精简版,量级是一两百行 JS,不是新依赖),加轨道相机的平移/旋转/缩放交互;
  拾取从"线性逆映射"换成"逆视图投影得到世界空间射线,和高度场求交"——这个
  求交完全可以在 JS 侧对 `parseTerrain` 已经拿到的原始高度数组做一次简单的
  步进射线求交,不需要 GPU 回读。

**结论(路线 A)**:这是三条路线里工程量最小、风险最低、唯一不作废种群、
唯一不碰 wire 协议的一条。真实工作量集中在 `web/render.js`(引入矩阵、把
地形从贴图 quad 换成高度场网格)和 `web/main.js`(相机交互 + 拾取求交)两个
文件,量级是"给一个 300 行的极简渲染器补上标准 3D 图形管线的基本部件",不是
从零设计。旧报告"笔记本核显都够"这个判断,复核后仍然成立——`32,258` 个三角形
和一次 `gl.POINTS` 调用对任何 2015 年之后的集成显卡都不构成压力。

### 2.2 路线 B:离散分层(2.5D)

#### 设计空间:两个子方案,代价差一个数量级

**B1——被动分层(最便宜)**:不新增任何 agent 可控的输出,只是让"你脚下是
地面还是树冠"变成由当前地形/位置隐式决定的一个状态,类似现在的
`terrain.forest`(林冠密度)已经在做的事——只是从"连续密度影响移动速度和视野
半径"变成"离散地跨越一条门槛,门槛两侧连到不同的食物场/捕食判定"。**如果
只做到这一步(不允许 agent 主动选择切层,分层完全由位置/地形决定),`in_dim`
可以不变、`out_dim` 可以不变、`genome_size` 不作废任何种群**——因为这本质上
只是给 `dynamics.graze`/`dynamics.predation` 读取的"有效地形"加了一层根据
位置计算出的离散判据,不需要额外感知它、也不需要额外的动作去控制它。这条子
方案买到的"3D 感"其实很有限:agent 感受不到"我在哪一层",行为也无法针对分层
做决策,只是资源分布和捕食可达性多了一条隐藏规则。

**B2——主动分层(agent 可以决定要不要爬树)**:需要(a)`out_dim` 增加一维
表示"爬升意图"(或者复用 `climb_cost` 的机制,把"移动进树冠格子"处理成一个
需要额外能量的动作而不需要新输出——这样可以不改 `out_dim`,代价是"进树冠"
不再是主动选择而是被动跟随移动方向进入林冠格子时自动触发,行为空间更受限);
(b)如果要让 agent **感知**自己在哪一层、周围哪一层有什么,至少需要给
`sensors.sense` 追加 1-2 个输入通道(自身层数 + 也许"树冠层食物"这个新维度的
retina 通道,类似 `peer` 通道当年被追加的方式,`[对应]` `sensors.py` 里
`peer_val` 追加在 memory 之后、不影响已有 slice 偏移的写法可以直接复用作
先例)。`[推导]` 一个具体的例子:追加 1 个"当前层"标量 + 8 个"树冠层食物在
各扇区"的 retina 通道,`in_dim` 从 67 涨到 76,若同时给 `out_dim` 加一维变成
3(见脚本核算,附录 A),`genome_size` 从 1381 涨到 1542(+11.7%)——**这已经
作废现有种群**,但涨幅远小于路线 C。

#### wire 协议

如果只是给 agent 状态加一个整数/浮点的 `stratum` 字段(用于渲染时区分"树上"
"地上"),`server/protocol.py` 的 agent 记录需要多一个字段——`[对应]`
`_HEADER`/`agents` 打包全部是位置字段,按 `CLAUDE.md` 的"append, never insert"
纪律,这是一次典型的"在末尾加一个字段"式改动:`_HEADER` 不用动,`agents`
每条记录从 20 字节变成 24 字节(多一个 f32),`web/main.js` 的 `STRIDE`/解析
偏移量需要同步加一列,但**不需要挪动任何已有偏移**——和 v6 加 `fruit_total`
时是同一类改动,风险可控。地形本身如果要表达"树冠层的独立地图"(而不是复用
现有 `forest` 密度场判定门槛),`UNTR` 消息也要加一张平面,同理是"追加"而不是
"插入"。

#### 渲染管线

如果分层只是逻辑状态、渲染上仍然把所有 agent 画在同一张 2D 图上(用颜色或
一个小图标区分"在树上"/"在地上"),渲染管线**几乎不用改**——这是 B1/B2
都可以选择的一个子决定,分层是仿真概念,不强制要求视觉上真的把它们摆到不同
高度画。如果想要视觉上体现"树上的点确实更高"(路线 A 的透视相机之上叠加分层
的视觉高度),那是路线 A 的相机工作 + 一个per-agent 高度 offset,量级很小。

#### 结论(路线 B)

**B1 完全不动 `in_dim`/`out_dim`/wire 协议,是三条路线里除了纯渲染之外最
便宜的一条,但买到的"3D 感"也最弱**——它只是给现有地形判据加了一条离散
门槛,агент 本身对"分层"这件事一无所知。**B2 才是名副其实的"2.5D"**,
代价是中等的 `in_dim`/`out_dim` 增长(附录 A 给出的例子是 +11.7% 的
`genome_size`)和一次 wire 协议的追加式改动,但仍然远比路线 C 便宜——空间
索引(`spatial.py`)、朝向表示、地形几何都不需要变成三维,只是多了一个离散
标量状态和一两个新的感知/动作维度。

### 2.3 路线 C:真 3D

这是唯一一条"名副其实"的 3D,也是三条路线里改动面最大的一条。下面按模块
逐一过一遍,而不是笼统地说"全部重写"——具体touch 面比想象中还要大,因为
2D 的假设渗透到了比"坐标只有两维"更深的地方。

#### `state.py`:形状变化不止 `pos`

- `pos: [n_max, 2] → [n_max, 3]`——这一条是直觉上就知道的。
- `heading: [n_max]`(一个标量弧度角)**无法**推广成三维朝向的标量表示。
  三维刚体朝向至少需要两个自由度(方位角+仰角)才能表示"看向哪个方向",
  如果还要表达"身体侧倾"则需要完整的三自由度(欧拉角或四元数)。这不是一个
  可以"顺手"解决的改动——欧拉角有万向锁,四元数的插值/mutation 在遗传算法
  语境下怎么变异是一个要单独设计的问题(对四元数的四个分量分别加高斯噪声
  再归一化,是否还保持"小扰动"的直觉?需要验证)。**这是路线 C 里第一个
  在其他两条路线里完全不存在的新问题**。
- `vel: [n_max, 2] → [n_max, 3]`。
- `memory: [n_max, slots, 3]`(`dx, dy, strength`)**必须**变成
  `[n_max, slots, 4]`(`dx, dy, dz, strength`)或等价的三维表示——`[对应]`
  `memory.py` 的 `advance`/`write`/`encode` 三个函数全部要重写,尤其是
  `encode` 现在把 `(dx, dy)` 转成 `(sin(bearing), cos(bearing))` 两个数
  (`memory.py:110-118`),三维下"方位"不再是一个标量角,要么用两个角
  (方位角+仰角各自 sin/cos,4 个数)要么用归一化方向向量(3 个数)+ 距离,
  每个记忆槽从 4 个特征变成 5-6 个,直接推高 `in_dim`。
- `hidden`、`genome`、`last_input`/`last_output` 形状随 `in_dim`/`out_dim`
  变化被动跟着变,不是独立的新问题。

#### `spatial.py`:分箱与邻居 gather 的维度提升

`[对应]` `build_table`(`spatial.py:27-50`)现在把 `pos_to_cell` 的结果
(`iy*g+ix`)当分箱依据;三维下变成 `iz*g*g+iy*g+ix`,分箱格子总数从
`sense_grid²` 变成 `sense_grid³`。`gather_neighbors`(`spatial.py:53-63`)
现在读 3×3=9 个相邻格子(`_OFFSETS`),三维下要读 3×3×3=27 个。`geometry`
(`spatial.py:66-79`)的 torus 环绕逻辑本身对维度数不敏感(`(d+half)%size-half`
逐分量做,天然推广),但候选列表从 `[n, 9K]` 变成 `[n, 27K]`。

**这一条不是靠读代码估计的,本次调研专门写了一个独立原型脚本实测了它**——
见下面第 3 节。结论先剧透:同样的 `k_neighbors`,3D 版本的分箱+邻居 gather
耗时是 2D 版本的约 **2.6-3.8 倍**(格子数匹配、也就是公平对比 27 个候选块
对 9 个候选块开销的版本),和候选张量列数 `27K/9K=3` 的解析比例吻合;如果
为了维持"每个格子平均装的 agent 数不变"(3D 更稀,格子必须更粗,否则同样
的物理体积被切成天文数字的格子)而不得不放大 `k_neighbors`,耗时和显存都
会随之进一步复合增长(K 翻倍,候选列数再翻倍)。

#### `sensors.py`:视网膜从平面扇区到球面分区

`[对应]` `sense`(`sensors.py:40-124`)现在把每个候选者的方位角
(`jnp.arctan2(delta[...,1], delta[...,0])`)离散化成 `retina_sectors=8` 个
平面扇区,每个扇区一个 `for s in range(R)` 的 Python 展开循环(`sensors.py:81-85`),
对 `food`/`prey`/`pred`/`water`/`slope`/`peer` 六个通道各做一次。三维下"方位"
变成球面上的一块面积(方位角 × 仰角的二维网格,或者更均匀的 icosphere/球面
细分),每个候选者需要额外算一个仰角、离散化到二维网格里,通道数从
`6×R` 涨到 `6×S`(`S` 是球面分区数)。给定同样的角分辨率,球面分区数天然比
平面扇区数多——一个粗略的类比是"纬度-经度网格":要达到平面 8 个扇区(45°
一格)的角分辨率,球面上大约需要 8 个方位 × 3-5 个仰角带,`S≈24-40`,`in_dim`
里这一块的贡献从 `48`(`6×8`)涨到 `144-240`,是 `in_dim` 增长的主要来源
(见附录 A 表格)。

`[现实,存在性经检索确认,仅作数量级旁证]` 球面感知离散化本身在生物/仿生
文献里有先例可以类比:仿生复眼传感器的报道给出的数量级是**数百到数千个**
离散感知单元才能覆盖近全向视场(例如一款约 1027 个"人工小眼"覆盖
1.5×1.5mm² 表面的复眼传感器,*Nature Communications*, 2022,
doi:10.1038/s41467-022-33072-8;另一款约 630 个小眼实现 180°×60° 的
柱面全景视场)。这条只是一个数量级参照,不是本节 `S≈24-40` 这个估计的
直接依据——真实复眼要覆盖的是连续视觉分辨率,Underworld 的球面分区是
离散的价值判断输入(食物/猎物/天敌/水/坡度/同类),两者的"分区数"没有
可比的换算关系,如实标注这条只是"生物学上球面覆盖需要很多离散单元"这个
方向性直觉的旁证,不要当成 `S` 应该取多大的证据。

#### `dynamics.py`:predation 的几何判据、移动模型都要重新推导

`predation`(`dynamics.py:136-184`)现在的判据是`dist < cfg.attack_range`
——一个平面圆盘内最近符合条件的猎物。三维下这自然推广成球体内最近猎物,
**几何上没有新问题**(`dist` 已经是欧氏距离,`spatial.geometry` 已经算好),
但**密度语义变了**:同样的 `attack_range`,3D 里"球体积"而不是"圆面积"
框住猎物,给定同样数量的个体撒在同样线度的空间里,三维下的有效遭遇率会
系统性下降——这是第 4 节要专门算的账,这里先记一笔:这个改动**在几何实现
上便宜**,但**在生态学后果上可能是全篇最贵的一条**。`act`(`dynamics.py:15-52`)
现在从 `outputs[:,0]`(转向)、`outputs[:,1]`(推力)算 `heading`/`vel`,
三维下如果朝向变成方位角+仰角两个自由度,`out_dim` 至少要从 2 涨到 3。

#### `terrain.py`:从一张高度场到体素场,或"高度场+垂直剖面"混合

这是本次调研认为**最容易被低估**的一块。`terrain.build`(`terrain.py:159-214`)
现在的全部字段都是 `[n_cells]`(`n_cells=grid²`)。如果真 3D 意味着"空气里
也能有障碍物、也能有食物层次"(比如林冠是一层悬浮在半空的资源,而不是地面
的一个属性),那地形就不再是一张高度场,而要么是一个完整的体素占据场
`[grid, grid, grid_z]`,要么退化成"一张 2D 高度场(地面)+ 每个 (x,y) 列上
几个离散的垂直层描述"(这其实已经很接近路线 B 的思路了,只是层数更多、
连续化程度更高)。选择纯体素化,内存和计算量都以 `grid_z` 再乘一个维度;
选择"高度场+垂直剖面",复杂度可控但那已经不是"路线 C 的最纯粹版本"而是
路线 B 的一个连续化变体。**这个选择本身是路线 C 里最大的一个未决设计问题**,
而不是一个实现细节。

#### `in_dim`/`out_dim`/`genome_size`:全线增长,作废现有全部种群

`[推导]`(公式见附录 A,可自行代入验证)取一个中等分辨率的球面分区数
`S=24`、记忆槽升级到每槽 5 个特征、`out_dim=3`(方位角速度、仰角速度、
推力):

| 配置 | `in_dim` | `genome_size` | 相对当前(1381)的倍数 |
| --- | --- | --- | --- |
| 当前 2D(`retina_sectors=8`) | 67 | 1381 | 1.0× |
| 路线 C,`S=12`(粗) | 95 | 1846 | 1.34× |
| 路线 C,`S=24`(中) | 167 | 2998 | 2.17× |
| 路线 C,`S=48`(细) | 311 | 5302 | 3.84× |

这张表的意义不在于"哪个 `S` 是对的"——`S` 本身是一个待决策的设计参数,不是
一个可以推导出唯一答案的量。意义在于:**无论选哪个 `S`,`genome_size` 都会
涨到当前的 1.3-4 倍**,而按 `CLAUDE.md` 的既定纪律,这意味着**当前演化出的
全部种群作废,`docs/trait_evolution.md`(体型基因、peer 通道)和
`docs/experiments.md`(踩踏两版实验)里记录的全部基线数字失效,必须重新演化、
重新建立基线**——这条代价和路线 A/B1 是质的不同,不是"贵一点",是"从零来过"。

顺带核实一下丢失报告里"1252→2596"这个数字:`[本世界实测]` 用当前公式反推,
`in_dim=59`(没有 `peer` 通道、`memory_slots=4×4=16`)、`trait_dim=2`(没有
`size` 基因)时 `genome_size` 确实精确等于 **1252**——说明旧报告这部分的
算术是认真做过的,不是拍脑袋。但反过来试图凑出 `2596` 这个数字时,在
`hidden=16` 不变的前提下无法用整数的 `in_dim` 精确凑出来(`in_dim≈143.1`,
非整数),**推测旧报告当时给的"2596"是一个粗略的"大约翻倍"估计,而不是
某个具体球面分区设计算出来的精确值**——这条本次调研没能完全对上,如实记录
这个差异,而不是硬凑一个 `S` 让它对上。本文档附录 A 的表格用当前的精确公式
重新给出了一组可验证的数字,建议以此为准。

#### wire 协议

`server/protocol.py` 的 `agents` 记录要多一个 `z` 分量(20 字节→24 字节),
`朝向`如果要发给客户端做渲染插值,也可能要多发 1-2 个字段。这仍然是"在
`agents` 记录末尾追加字段"式的改动,不违反"append, never insert"纪律,但
`UNTR` 地形消息如果要表达体素或多层地形,需要一次结构性更大的改动(不再是
"三张 `grid²` 的 u8 平面",可能是 `grid²×层数` 或完整的 `grid³` 体素平面)
——这不是"加个字段"就能完成的,是消息格式本身的重新设计。

#### 渲染管线

路线 C 的渲染需求和路线 A 高度重叠(都需要透视相机、都需要从地形数据生成
几何),但如果地形变成体素/多层结构,`web/render.js` 需要的不再是"一张高度场
生成一张网格",而是某种体积渲染或者按层分别生成网格再合成——工程量比路线 A
纯做透视相机 + 高度场网格明显更大,虽然仍然远小于仿真核心的改动量。

#### 结论(路线 C)

真 3D 在**每一个模块**都有实打实的工作量:`state.py`(姿态表示是一个新的、
其他路线不存在的难题)、`spatial.py`(索引维度提升,已实测 2.6-3.8 倍开销)、
`sensors.py`(球面感知离散化,`in_dim` 涨 1.3-4 倍)、`dynamics.py`(捕食
几何简单但生态学后果可能致命,见第 4 节)、`terrain.py`(体素化是一个未决的
架构级设计问题,不是实现细节)、`memory.py`(记忆槽结构性改变)、wire 协议
(agent 记录追加字段可控,地形消息需要重新设计)、渲染管线(比路线 A 更重)。
**没有一条是不能做的,但没有一条是便宜的**,而且作废现有全部种群这件事,
按 `CLAUDE.md` 的既定纪律,意味着 `docs/trait_evolution.md`、
`docs/experiments.md` 里已经跑出来的全部长种子实验结论都要重新建立基线才能
继续引用。

## 3. 单卡够不够:实测先行,推导补空白

### 3.1 2D 基线:steps/s 与显存,实测

`[本世界实测]` 用 `scripts/run_headless.py` 和独立的 JAX benchmark 脚本
(见附录 B 的方法论)在这台 RTX 4090 上测的当前 2D 默认配置
(`n_max=16384, grid=128, retina_sectors=8, k_neighbors=24, sense_grid=24,
genome_size=1381`):

| 指标 | 数值 | 说明 |
| --- | --- | --- |
| 稳态 steps/s(无显卡争用) | **~590** | 多次独立测量收敛到 587-593 这个区间 |
| 稳态 steps/s(有其他 agent/训练任务争用) | **~40-160** | 视争用强度,同一份代码同一个配置 |
| 显存峰值(`n_max=16384`,`jax`短时段测量) | **~357-405 MiB** | 与 `CLAUDE.md` 记录的"918 MiB,长跑测量"同量级,差异来自测量窗口长短(genome 历史、优化器状态等在更长的 run 里会进一步分配) |
| 编译时间(首次 `jax.jit`) | **~7-9 秒** | 一次性,不计入稳态 |

**关键的方法论发现**:`[本世界实测]` 每个配置编译完成后的**第一个** chunk
稳定地比后续 chunk 慢 10-15 倍(比如 41→591 steps/s),后续 chunk 才稳定在
真实稳态。原因不明确——可能是 GPU 频率从省电状态爬升到 boost 频率需要时间,
也可能是某种 XLA/cuBLAS 一次性初始化开销没有被 `jax.block_until_ready`
之外的编译计时捕捉到。这条本身不影响 3D 可行性判断,但如果之后有人用这份
代码库做性能对比,**必须像这次一样丢弃编译后的第一个 chunk**,否则会把一个
测量伪影错当成真实性能差异。

几个可以从这张表直接读出、且不太受争用噪声干扰的**结构性**结论(争用会改变
绝对值,但不太会翻转相对大小关系):

1. **`n_max` 和 `k_neighbors` 是当前系统里对性能/显存最敏感的两个旋钮**,
   两者都直接决定邻居候选列表 `[n, 9K]` 或类似形状张量的大小。`retina_sectors`
   紧随其后,因为通道数和 `in_dim`(进而 `genome_size`、`brain.forward` 的
   矩阵乘法开销)直接挂钩。
2. **`grid`(平面场的分辨率)在测试范围内(64-2048)几乎不影响 steps/s 和
   显存**——`plant`/`fruit`/`trample`/地形的全部字段加起来,按显存峰值反推,
   边际成本约 **77 字节/格**(`[推导]`:`grid=512→2048` 之间,显存从
   374.2 MiB 涨到 662.0 MiB,增量 287.8 MiB 对应增量 393 万格,
   `≈76.8 字节/格`,包含了 `lax.scan` carry 里的字段双缓冲和一些临时数组
   开销,不是单纯的"8 个 `f32` 数组 = 32 字节"这种理论下限)。这条对 3D
   化很关键:它说明**"格子数量本身"目前不是瓶颈**,瓶颈更可能出现在
   "agent 数 × 邻居候选数"这一侧,而不是地形/资源场那一侧——除非 3D 化
   之后格子数暴涨到远超本次测试的上限(见 3.3)。
3. **`sense_grid` 几乎完全不影响性能或显存**——这是符合代码结构的:
   `sense_grid` 只决定分箱的粗细,`build_table` 的输出形状是
   `[n_sense_cells, K]`,`n_sense_cells` 增大只是让每个格子里装的 agent
   更少,真正的候选列表大小由 `9K`(3×3 块 × 每格 K 个)决定,和
   `sense_grid` 本身脱钩。这条对 3D 化的含义是:**分箱粒度不是一个昂贵的
   旋钮,可以放心调粗调细去匹配 3D 下的密度,不用担心这本身产生额外开销**
   ——真正要担心的是 3D 下每个 agent 要读 27 个格子而不是 9 个,以及为了
   不产生静默 overflow(`CLAUDE.md` 提到过:格子里 agent 数超过 `k_neighbors`
   就会被静默丢弃,同时影响视觉和捕食)而不得不放大的 `k_neighbors`。

### 3.3 3D 空间索引:独立原型实测(子 agent 结果)

上面这些都是 2D 代码的实测。3D 化里最核心、最容易低估的一块是空间索引本身
——从"3×3=9 个候选格子"变成"3×3×3=27 个候选格子",这条本次调研专门派了
一个子 agent 写独立原型脚本(不改 `underworld/spatial.py`,只是复刻同一套
"分箱→gather→torus 相对几何"逻辑到一个脱离仓库的 JAX 脚本里)去实测,结果:

`[本世界实测,原型代理测量,非仓库代码路径]`:

- **同样的 `k_neighbors`,3D 版比 2D 版慢约 2.6-3.8 倍**(在"格子总数匹配"
  这个公平对比设置下——即让 `sense_grid_3d³ ≈ sense_grid_2d²`,这样才是
  真正对比"27 个候选块 vs 9 个候选块"本身的开销,而不是顺带对比了格子总数
  变化带来的影�响)。这个倍数在争用轻重不同的多轮测量里都稳定收敛在这个
  区间,并且与候选张量列数的解析比例 `27K/9K=3` 吻合得很好。
- 如果不做"格子总数匹配"(即保持 `sense_grid` 数值不变,让 3D 的格子总数
  变成 `sense_grid³`,远超 2D 的 `sense_grid²`),测出来的倍数反而更小
  (~1.4-2.1 倍)——这是因为格子变多、每格更稀疏,`argsort`/`cummax` 这类
  分箱操作的常数项占比变化。**这条差异提醒一件事:"3D 慢多少"这个问题
  没有单一答案,取决于分箱粒度怎么选**,而分箱粒度本身又是一个要匹配
  "每格 agent 数不能超过 `k_neighbors`、否则静默丢弃"这条约束的设计决策。
- **显存**:候选邻居张量按设计精确是 3 倍(`27K` 对 `9K` 列),实测 GPU
  峰值在不同 `n_max` 下的 3D/2D 比值落在 **3.3-4.0 倍**(略高于理论 3 倍,
  来自额外中间缓冲的固定开销)。如果为了维持"每格平均 agent 数不变"(3D 更稀,
  格子必须更粗,否则需要天文数字的格子)而把 `k_neighbors` 从 24 放大到
  48 或 96,代价复合增长:**耗时约 2D 基线的 3.3-5 倍(K=48)到 4.7-7.3 倍
  (K=96),显存约 7.9 倍(K=48)到 13.5 倍(K=96)**——因为候选列数
  `M=27K` 随 `K` 线性增长,而这条放大在 3D 里几乎是必需的,不是可选的
  优化(否则大量近邻会被静默丢出视野和捕食判定之外)。

**这一段的限定必须原样保留,不能在后续引用里丢掉**:以上数字来自一个
**脱离仓库、只测分箱+gather+相对几何这一核心操作的独立原型脚本**,不包含
`diet`/predation 等生态逻辑,不是对 `underworld/spatial.py` 真实代码路径的
测量。它测的是"3D 化空间索引这一个模块的代价下限",而不是"整个 world_step
3D 化之后的总代价"——`sensors.sense` 里的 retina 分箱循环(`sensors.py:81-85`)
和 `dynamics.predation` 里对候选列表取 `argmin`(`dynamics.py:158`)都直接
建立在同一个候选列表 `M` 之上,3D 化后 `M` 变大,这两处的开销会按类似比例
跟涨,而 `brain.forward` 的开销随 `in_dim` 增长(附录 A 表格里 1.3-4 倍不等)
是另一路独立的开销来源,不包含在这个原型测量里。**把这几路加起来**,一个
合理但仍是外推而非实测的估计是:**整条 `world_step` 3D 化后的总体减速大概率
落在 3-8 倍这个区间**,与丢失旧报告那个打了 CPU 微基准折扣的"6-10 倍"在
量级上是一致的、经得起交叉验证的,但这次的下界(空间索引本身的 2.6-3.8 倍)
是本机 GPU 上真实测出来的,而不是转译自一个不同硬件上的微基准。

### 3.4 地形场:`grid³` 在什么分辨率下还放得下 24 GB

`[推导]`,基于 3.2 节实测的 `~77 字节/格` 经验常数:如果 3D 地形延用现有的
"每个格子挂 8-11 个 `f32` 标量场"这套设计(`height`/`grad_x`/`grad_y`/
`water_dist`/`forest`/`rock`/`capacity`/`fruit_capacity` 加上仿真态里
`plant`/`fruit`/`trample` 三个动态场),把这套场铺进一个 `grid³` 的三维网格:

| 3D 网格边长 `g`(各向同性,`g³` 个格子) | 格子总数 | 按 77 字节/格估算的显存 |
| --- | --- | --- |
| 128 | 2,097,152 | ~161 MB |
| 256 | 16,777,216 | ~1.29 GB |
| 384 | 56,623,104 | ~4.36 GB |
| 512 | 134,217,728 | ~10.3 GB |
| 600 | 216,000,000 | ~16.6 GB |

`[推导]` **各向同性(x/y/z 同一分辨率)的 3D 网格在 `g≈500-600` 附近就会
吃掉 24 GB 里的大半**,这还没算 agent 侧的数组(`n_max=16384` 时约
0.4-1.1 GB,视测量窗口)和 JAX/XLA 自身的工作区开销(scatter-add、argsort
这类操作在 XLA 里经常需要几倍于数据本身大小的临时缓冲)。**但这张表本身
基于一个值得质疑的假设:z 方向真的需要和 x/y 一样精细吗?** 当前世界
`world_size=512`,而生态学上真正有意义的垂直结构(林冠高度、地面到树冠的
层次)大概率是一个远小于 512 的尺度——项目自己的 `vision_radius=21`、
`attack_range=6.0`、`food_sample_dist=9.0` 这些**agent 尺度的绝对长度**
都在个位数到二十几个世界单位这个量级,不是几百个单位。**如果 z 方向只用
一个小得多的分辨率(比如 16-64 层,对应一个几十个单位高的"林冠带"而不是
一个和世界等宽的立方体),`grid_xy² × grid_z` 这种各向异性网格能把地形场
的显存开销压低一到两个数量级**,和各向同性立方体比,这不是"锦上添花"的
优化,而是"要不要把显存预算浪费在一个物理上没有意义的维度"这个更基本的
设计问题——第 4 节的生态学推导会进一步说明,z 方向的物理尺度选择还直接
决定捕食遭遇率会被稀释多少,是一个牵一发动全身的参数,而不是渲染细节。

**结论(3.4)**:`grid³` 本身,只要 z 方向的分辨率/物理尺度选得合理(不是
"和 x/y 同宽同高的立方体"这种默认假设),不会是 24 GB 显存下的硬约束——
真正的约束在下一节:**在任何显存放得下的分辨率下,3D 化会不会先把捕食者
从生态上逼灭绝**。

## 4. 生态学后果:这可能比性能更致命

用户的问题本身已经预判了这一节的重要性:"如果 3D 化必然让捕食者灭绝,那
'技术上可行'就是句空话"。这一节要正面回答这个问题,不能只给一个"遭遇率
下降到 0.15 倍"这样的裸数字然后带过——本次调研的核心工作之一就是把这个
数字从"记不清楚的口头转述"变成一条可以逐步核对的推导。

### 4.1 遭遇率如何随维度变化:从头推导

`[推导]` 一个捕食者以相对速度 `v` 移动、探测/攻击半径为 `r`,在单位时间内
"扫过"的空间量,在二维平面里是一条宽度 `2r`、长度 `v` 的带状区域,面积
`2rv`(单位:长度²/时间);在三维空间里是一个半径 `r`、长度 `v` 的圆柱体,
体积 `πr²v`(单位:长度³/时间)。这正是经典"气体动理论式"遭遇率模型的核心
——扫过的量乘以对应维度下的个体密度,就是遭遇率:

```
2D:  Z_2D = 2 r v σ_2D,   σ_2D = N / L²   (N 个个体撒在边长 L 的平面上)
3D:  Z_3D = π r² v σ_3D,  σ_3D = N / (L² · L_z)  (同样 N 个个体撒进一个 L×L×L_z 的体积)
```

`L_z` 是第三维的物理尺度,不必等于 `L`——这一点在下面会看到是全篇最关键
的一个自由参数。两式相除:

```
Z_3D / Z_2D = (π r² v σ_3D) / (2 r v σ_2D) = (π r / 2) / L_z
```

注意 `N` 和 `L`(世界的水平尺度)在这个比值里**完全消掉了**——遭遇率的
维度依赖只取决于攻击半径 `r` 和第三维的物理尺度 `L_z`,和种群规模、世界
水平大小无关。代入当前配置 `attack_range=r=6.0`:

| 假设的第三维尺度 `L_z` | `Z_3D / Z_2D` | 相对现实感的粗略类比 |
| --- | --- | --- |
| 512(和世界水平尺度一样高的立方体) | **0.018** | 明显不合理——没有生态系统的"天空"和地面一样宽 |
| 64(比如一片高大乔木林,林冠~60余单位高) | **0.147** | 与旧报告口头转述的"0.15 倍"几乎完全吻合 |
| 32 | 0.294 | 较矮的森林/灌木层 |
| 16 | 0.589 | 很浅的"准二维"世界 |
| 6(等于攻击半径本身) | 1.571 | 退化到"3D 和 2D 差不多",符合边界直觉——世界薄到攻击半径的尺度,3D 体积其实退化成一个 2D 圆盘乘一个薄层,遭遇率不应该被显著稀释,算出来确实回到了 O(1) |

**这张表是本节最重要的发现,值得停下来读一遍**:0.15 倍这个数字**不是
凭空冒出来的**,它对应的是"选一个约 64 个世界单位高的第三维"这个具体假设
——旧报告的口头转述如果真的说过 0.15,那么无论有没有明说,它隐含地做了
一个"这个世界大概是这么高"的假设。而这个假设**从未被写下来讨论过是否合理**
,这正是"口头转述丢失论证过程"的典型后果:结论被记住了,推导它的假设消失了。
反过来说,**这也是本次调研认为最重要的一条方法论收获**:3D 化对生态的
冲击程度,不是一个由"3D 本身"决定的常数,而是**由第三维选多"薄"这个设计
决策决定的连续量**——这个参数(可以类比为 `world_size_z` 之类的新
`Config` 字段)在真正实现路线 C 之前必须被显式地讨论和选定,而不是隐含地
被某个人的直觉选中。

### 4.1.1 文献核实:上面的推导不是自说自话

派出去专门核实这类模型的子 agent 带回了比预期更扎实的结果,直接支持上面
从"扫过体积×密度"量纲分析独立做出的推导:

- `[现实,已核实全文]` **Pawar, Dell & Savage (2012), "Dimensionality of
  consumer search space drives trophic interaction strengths", *Nature*
  486:485-489, doi:10.1038/nature11131**。这篇论文给出的一般化搜索率公式
  正是 `α = s_D · v_r · d^(D-1)`,其中 `D` 是搜索空间维度、`d` 是反应/
  探测距离、`v_r` 是捕食者-猎物相对速度、`s_2=2`(二维,来自"宽度 `2d`
  ×速度"扫过的矩形面积速率)、`s_3=π`(三维,来自半径 `d` 的圆柱体扫过
  体积速率 `πd²v`)——**和本节独立推出的 `2rv`(2D)/`πr²v`(3D)逐项
  对应,是同一个模型,子 agent 核实的文献给出了这个模型在生态学文献里
  的正式出处和一般化形式**。
- 论文原文的一句话直接、明确地背书了 4.2/4.3 节要论证的结论,值得整句
  引用:*"If resources had the same numbers (but not densities) in 2D and
  3D habitats (for example, 1 kg m⁻² and 1 kg m⁻³), resources would
  probably be too sparse for a 3D search space to be advantageous."*
  ——这正是 Underworld 的处境:如果 3D 化只是给同样数量的个体多加一个
  自由度、而不特意把密度按体积重新标定,论文原作者自己给出的预期就是
  "太稀疏,3D 搜索空间不划算"。论文正文的头条结论(现实里 3D/浮游生境
  的捕食速率通常高于 2D/陆地生境)**不能反过来引用**——那个结论成立的
  前提是真实物种在 3D 生境里的密度天生比 2D 生境高 2-3 个数量级,和
  "同样个体数塞进同样绝对尺度的体积"是两个不同的比较,子 agent 的报告
  特别提醒了这一点,这里如实转达,避免断章取义。
- `[现实,存在性已核实,具体公式文本未核实]` **Gerritsen, J. & Strickler,
  J.R. (1977), "Encounter Probabilities and Community Structure in
  Zooplankton: a Mathematical Model", *J. Fish. Res. Board Can.* 34(1):
  73-82, doi:10.1139/f77-008**——这是记忆中最先想到的那篇经典文献,子
  agent 确认了作者、年份、期刊、DOI 均真实存在,是 Pawar et al. (2012)
  这套一般化模型的原始特例(浮游动物、反应距离、相对速度导出体积扫掠率）。
  **但原文的精确公式文本没有拿到**(数据库访问受限),网络上一个二手转述
  给出的具体系数**没有核实来源,不应采信其系数细节**——只把"3D 是体积
  扫掠、和反应距离的平方成正比"当作可信的结构性事实,这条有 Pawar 2012
  原文兜底。
- `[现实,存在性已核实,内容仅部分核实]` **Gurarie, E. & Ovaskainen, O.
  (2013), "Towards a general formalization of encounter rates in
  ecology", *Theoretical Ecology* 6:189-202**——提供了一个更一般的遭遇率
  理论框架(首次遭遇率 vs 平均遭遇率,依赖运动行为、感知尺度、目标分布),
  子 agent 没能确认它是否显式给出了 2D/3D 的公式差异,**只作为"存在更
  一般理论框架"的旁证列出,不作为具体数值的来源**。

**用 Pawar (2012) 的公式重新代入 Underworld 自己的参数**(`d=attack_range
=6.0`,世界水平尺度 `L=world_size=512`,`D=3` 时用各向同性假设
`ρ_3D=N/L³`,即 4.1 节表格里"`L_z=512`"那一行的等价情形):

```
比值(3D/2D) ≈ (s_3/s_2) · (d/L) = (π/2) · (6/512) ≈ 0.0184
```

这与本节 4.1 表格独立算出的 `0.018` 完全一致(两条推导路径互相印证:
一条是本次调研自己从"扫过体积×密度"做的量纲分析,一条是子 agent 核实到
的、已发表 40 多年、并在 2012 年被 *Nature* 重新一般化的经典模型),**两
到三个数量级的下降是这个各向同性假设下的量级**,而 4.1 节已经说明这个
"和世界一样高的立方体"假设本身不合理——真正有意义的问题不是"3D 会不会
稀释遭遇率"(会,而且有独立推导和已发表文献双重印证),而是"选一个多
`浅`的第三维,把稀释压到可以接受的程度",这是 4.2/4.3 节要继续算的账。

### 4.1.2 一个诚实的空白:人工生命领域没有查到直接先例

子 agent 专门找过"有没有已发表的人工生命/演化模拟系统,报告过'从 2D 扩到
3D 导致种群密度/遭遇率/存活率下降'这个具体现象",**结果是没有查到**,如实
记录这个空白,而不是暗示"业内早有共识":

- Karl Sims, *"Evolving 3D Morphology and Behavior by Competition"*,
  Artificial Life IV Proceedings, MIT Press, 1994, pp.28-39——`[现实,
  存在性已核实]`,3D 物理仿真+竞争式演化的经典先例,但这是原生 3D 设计,
  没有对照的 2D 版本可比较,**不能**当作"迁移代价"的证据,只能当作
  "3D 演化仿真本身是可行的"这条更弱的旁证。
- Bert Wang-Chak Chan, *"Lenia and Expanded Universe"*, arXiv:2005.03742
  (2020)——`[现实,存在性已核实]`,确认 Lenia 元胞自动机在 3D 下依然
  能产生稳定的类生命形态,但讨论的是模式本身能否存在,不涉及基于个体
  的捕食者-猎物遭遇率或种群密度问题,**和本文档的关切不是一回事**。
- Larry Yaeger 的 *Polyworld*(记忆中可能是"世界本身是 2D、只做 3D
  渲染"的先例,与本文档路线 A 高度类似)——`[待核实]` 子 agent 的检索
  配额已耗尽,**这一条完全没有核实,只是调研过程中浮现的记忆,不应采信,
  如果之后需要应单独核实**。

**如实的结论是**:本次调研没能找到任何一篇论文正面报告过"2D 迁移到 3D
导致捕食者/遭遇率下降"这个具体现象——这既不是"文献证明这不会发生",
也不是"文献证明这会发生",而是这个具体问题(在一个人工演化生态系统里,
把固定数量的个体从平面搬进体积,遭遇率怎么变)**似乎没有被 ALife 领域
专门测量和报道过**。本文档 4.1/4.1.1 给出的定量论证,是从生态学的一般
遭遇率理论(Pawar 2012 已发表、已核实)推导到 Underworld 这个具体系统的
一次**外推**,不是对某篇"别人已经在类似系统里测过"的复述——这个区别
必须在引用这份文档时保留。

### 4.2 对照已知的近阈值生态:这个稀释足以杀死捕食者吗

这是本节要正面回答的问题。已经确认的经验事实(`[本世界实测]`,
`docs/experiments.md`/`docs/mortality.md` 记录,本次调研重新核对过其
存在但没有重新重跑那些长种子实验,原始结论直接引用):

- `carn_cost` 从 `0.15` 降到 `0.10` 才能让捕食者在 20000 步内不灭绝
  ——`[对应]` `config.py` 里 `carn_cost` 字段的注释明确写着"0.15 时在
  每个测试种子上都灭绝了"。这不是一个"调一下能救回来"的松弛参数,是一个
  在当前 2D 几何和当前捕食判据下已经被反复验证过的、狭窄的生存窗口。
- `n_init` 的甜区在两个方向都很窄——离平衡点太近,捕食者奠基种群太小,
  自身随机性就能灭绝;离平衡点太远,初始崩溃本身杀光捕食者(7 倍崩溃在
  测试过的每个种子上都团灭了捕食者)。
- 死因分解(`docs/mortality.md`)显示捕食只占死亡的 9.95%(反事实下界)
  到 16.6%(上界)——本身已经是一个不算大的份额,任何进一步压低捕食
  发生率的改动,都是在一个已经不算稳固的份额上继续施压。

把 4.1 节的推导和这些已知事实放在一起看:**在任何"合理"的第三维尺度选择
下(64 甚至 32 个世界单位,而不是和世界一样宽的 512),遭遇率会被稀释到
现有值的 15-30%。给定捕食者生存已经是一个 `carn_cost` 从 0.15 降到 0.10
——降低 33%——才勉强稳住的近阈值系统,一个让核心资源(猎物遭遇机会)
下降 70-85% 的改动,在不同时调整其他补偿参数的前提下,几乎可以确定会
让捕食者从系统里消失**。这不是一个"可能"的风险,是把 4.1 节的定量结果
和 `docs/mortality.md`/`config.py` 里已经验证过的敏感度直接相乘得到的
结论。

### 4.3 正面回答:路线 C 是否"技术上可行但生态学上是空话"

**是,除非同时对冲这个稀释**。3D 化(路线 C)本身在工程上是可以做的
(第 2.3 节),显卡也放得下(第 3.4 节),但如果不做任何补偿,几乎可以
确定会杀死捕食者种群,那样一来"我们支持 3D 了"这句话在生态学意义上就是
空的——种群会退化回纯食草者,捕食-猎物这一整套已经调好的机制形同虚设。

**可能的对冲手段**,每一条都有自己的代价,没有免费的:

1. **选一个"浅"的第三维**(`L_z` 远小于 `world_size`)。这是最直接、最
   便宜的对冲——4.1 节的表格显示,`L_z` 越接近 `attack_range` 量级,
   稀释效应越接近于无。但这也意味着"真 3D"退化成"一个很扁的准 3D",
   离用户想要的"真正的立体世界"有多远,是一个需要明确说清楚的权衡,而
   不是悄悄选一个数字然后不提。
2. **放大 `attack_range`/`vision_radius`**。4.1 节的公式里 `Z_3D/Z_2D`
   正比于 `r`,把攻击范围放大能部分抵消稀释,但这本身会改变已经调好的
   捕食生态(`attack_range=6.0` 现在的设定理由之一就是"够短,给猎物一个
   体型庇护所",`config.py` 注释原文;放大它意味着重新打开一整套已经
   调过的平衡)。
3. **提高种群密度**(增大 `n_max`/`n_init`,或缩小世界)。这条直接把
   第 3 节的性能/显存代价重新拉回来——遭遇率公式里 `σ_3D = N/(L²L_z)`,
   提高 `N` 或缩小 `L` 都能推高密度,但 `n_max` 更大直接意味着更大的
   邻居候选列表(3.3 节实测的 3-7 倍量级代价会再乘上密度提升的倍数)。
4. **降低 `carn_cost` 或提高 `pred_rate`/`meat_water_frac`**,直接从
   代谢侧而不是遭遇率侧补偿。这是最"便宜"的旋钮(不改变任何形状,纯参数
   调整),但按 `CLAUDE.md` 的既定纪律,任何生态参数改动都需要六种子配对
   验证,不能凭一次运行下结论,而且"补偿到刚好活下来"这件事本身可能需要
   反复试探,不是一次调参能确定的。

**如果要做路线 C,这四条里至少要认真做第一条(选一个物理上合理的浅
`L_z`),并且在正式跑长种子实验前,先专门做一次"3D 遭遇率是否真的按
4.1 节的公式稀释"的小规模验证**(比如固定种群、关闭繁殖/死亡,只测
"给定同样的捕食者-猎物初始分布,3D 版本单位时间内记录到的捕食判定次数
是不是确实落在推导的比例区间内")——这是一个比直接跑 20000 步长种子
便宜得多的证伪实验,应该作为路线 C 立项前的第一个动作,而不是先把整个
仿真核心重写完再发现遭遇率对不上。**这个验证实验已经做了,见 4.4 节。**

### 4.4 验证实验:4.1 节的公式测出来了吗

这一节回答 4.3 节末尾提议的问题,不再是推导,是**实测**。三个独立原型
(不改 `underworld/` 任何文件,全部在会话 scratchpad 完成,不提交仓库)
分别测了三个层次:纯几何(均匀分布)、纯几何(真实分布)、真实移动下的
捕食判定发生率。方法与全部数字见附录 C。

#### 实验 1:均匀分布下的静态遭遇计数,`N` 用真实配置

`[本世界实测]` 在 2D/3D 环面上撒 `N` 个均匀随机点,统计"环面距离 <
`attack_range`(=6.0,`config.py` 真实值,不是编的数字)的有序对"数量,
每格配置独立重复 24 次(`n_max=16384` 因为单次 `O(N²)` 距离矩阵接近这台
显卡当时的可用显存,重复 6 次)取均值。`N` 分别取 `config.py` 的
`n_init=2000`、`n_max=16384`,以及一次真实 4000 步 headless 跑到近稳态后
的实际存活数 `n_eq=1041`(`carnivore_frac=0.126`,与 `docs/mortality.md`
记录的量级一致)——**三个 `N` 测出的 `Z_3D/Z_2D` 比值几乎完全重合**
(比如 `L_z=64` 时分别是 0.1282、0.1266、0.1195,`L_z=16` 时是
0.505、0.4994、0.4933),偏差都在 6% 以内,直接验证了 4.1 节推导里
"`N` 和世界水平尺度在比值里完全消掉"这条数学结论——这不是一个近似,是
测出来精确成立的。

| `L_z` | `Z_3D/Z_2D`(实测,三个 `N` 的代表值) | 4.1 节公式 `(πr/2)/L_z` | 实测/公式 | "静态占据"模型 `(4r/3)/L_z` |
| --- | --- | --- | --- | --- |
| 512 | 0.0138–0.0157 | 0.0184 | 0.75–0.85 | 0.0156 |
| 64 | 0.1195–0.1282 | 0.1473 | 0.81–0.87 | 0.1250 |
| 32 | 0.2444–0.2586 | 0.2945 | 0.83–0.88 | 0.2500 |
| 16 | 0.4933–0.5050 | 0.5890 | 0.84–0.86 | 0.5000 |
| 8 | 0.8248–0.8629 | 1.1781 | 0.70–0.73 | 1.0000 |
| 6 | 0.8737–0.9194 | 1.5708 | 0.56–0.58 | 1.3333 |

**这张表回答了 4.3 节提的问题:公式在函数形式上被证实,在量级上被证实,
在精确常数上被证伪**。测出来的比值和 4.1 节公式 `(πr/2)/L_z` 几乎逐行
对应同一个 `1/L_z` 标度,量级完全一致,但精确数值系统性地落在公式的
**73%–88%**(`L_z≥16` 时),而几乎精确落在另一个解析模型上——`(4r/3)/L_z`,
这是"静态占据"模型(用disk 面积/ball 体积算某一瞬间有多少个体落在半径
`r` 内),而不是 4.1 节引用 Pawar (2012) 的"扫掠"模型(移动搜索者单位时间
扫过的圆柱体积)。**这不是实验做错了,是两个模型本来就在测不同的量**:
实验 1 做的是"任意时刻有多少对距离小于 `r`"的静态快照统计,这精确对应
"静态占据"模型;4.1 节的公式对应的是"移动搜寻者单位时间新进入探测范围
的目标数"这个动态通量。两者标度形式相同(都正比于 `1/L_z`,`N` 和 `L`
都消掉),差的是一个约 15% 的几何常数(`4/3` 对 `π/2`,比值 0.849,和
实测的 0.85 左右吻合得很好)。`L_z≤8` 时两个模型都开始失真(实测比值
明显低于两者),原因是 `L_z` 已经小于或接近 `attack_range`,环面 z 方向
的 wrap 让"球体积不重叠"这个模型假设本身失效——4.1 节表格自己在
`L_z=6` 那一行也承认"退化到和 2D 差不多",这里的实测确认了那个退化区间
从 `L_z≈8` 附近就已经开始,不是恰好在 `L_z=6`。

#### 实验 2(加分项):真实空间分布下,比值是否还一样

`[本世界实测]` 用同一次 4000 步 headless 跑(`n_eq=1041`)的**真实**
`x/y` 位置(食草者散布内陆、肉食者贴河,不是均匀分布)做 x/y,`z` 仍然
均匀随机,重复 24 次每个 `L_z`。两个发现:

1. **真实分布的绝对遭遇计数比均匀分布高约 1.8 倍**(`Z_2D`:真实 0.855
   对均匀 0.463,同样 `N=1041`)——这本身是一条独立于 3D 化的发现:agent
   聚集在水源/森林附近,局部密度显著高于"把同样数量的点撒进同样面积"这个
   零假设,与 `CLAUDE.md` 记录的"`inland_frac` 比随机撒点低得多"这条空间
   聚集性证据方向一致。
2. **但 `Z_3D/Z_2D` 这个稀释比值本身,在真实分布下和均匀分布下几乎一样**
   ——多数 `L_z` 上偏差在 10% 以内(`L_z=512` 相差 5.0%,`L_z=32` 相差
   -1.4%,`L_z=6` 相差 7.5%),只有 `L_z=384` 一行出现 36% 的偏差,更像是
   `N_Z_SEEDS=24` 在那个具体格点上的采样噪声,而不是系统性效应(相邻的
   `L_z=256/192` 两行偏差只有 8.4%/1.0%,不支持"384 附近有什么特殊物理"
   这个解释)。**结论**:agent 聚集在哪里(水边、内陆)会改变遭遇的绝对
   频率,但不会改变"3D 比 2D 稀释多少"这个比例——稀释是几何(`r`、`L_z`)
   决定的,和种群的空间分布模式基本无关。这条只用了一次真实分布快照,
   没有跨种子重复,达不到 `CLAUDE.md` 六种子的证据标准,当加分项的旁证
   看待,不当作独立确证。

#### 实验 3:真实移动 + 真实食性过滤下的捕食判定发生率

`[本世界实测]` 这是三个实验里最接近真实 `dynamics.predation` 语义的一个:
固定种群(同一次 headless 跑的 `n_eq=1041` 真实 `alive`/`diet`,关闭
繁殖/死亡),复刻 `predation()` 的判据 `eligible = valid & (d_i - d_j >
diet_delta) & (dist < attack_range)`(`diet_delta=0.15`,均取 `config.py`
真实值;`e_j > 0` 简化为恒真,因为没有采集真实 `energy` 场,这是一个如实
披露的简化,只在能量瞬时接近零的边缘情况下才会有偏差),用持续随机游走
(朝向/方向逐步高斯扰动,2D 是角度、3D 是单位方向向量,速度取
`0.5×max_speed=4.0`,两种食性同速——这本身是简化,真实食草者/肉食者
速度不同,但同速能避免"食性速度差"和"维度稀释"这两个效应在对比里混在
一起)跑 300 步,每步用 `argmin` 取每个 agent 的最近合格猎物(严格复刻
真实代码"每个捕食者只咬最近一个合格猎物"的语义),累加"有合格猎物"的
agent 数。

| `L_z` | 3D 判定总数(300 步) | 判定比值(3D/2D) | 4.1 节公式预测 |
| --- | --- | --- | --- |
| 512 | 401 | 0.0270 | 0.0184 |
| 128 | 1,380 | 0.0929 | 0.0736 |
| 64 | 2,979 | 0.2006 | 0.1473 |
| 32 | 5,033 | 0.3389 | 0.2945 |
| 16 | 9,291 | 0.6255 | 0.5890 |
| 8 | 15,388 | 1.0360 | 1.1781 |
| 6 | 14,727 | 0.9915 | 1.5708 |

(2D 基线:14,853 次判定 / 300 步 = 49.51 次/步。)

在真实移动下,测出的比值反而**略高于** 4.1 节的公式(`L_z=512` 时实测
是公式的 1.47 倍,`L_z=64` 时是 1.36 倍),而不是像实验 1 那样系统性偏低
——这是意料之中的:移动的 agent 确实在"扫过"空间,更接近 4.1 节假设的
"移动搜寻者"模型,而不是实验 1 的单帧静态快照。三个独立方法(静态均匀、
静态真实分布、动态真实移动+食性过滤)测出的比值全部落在 4.1 节公式的
**0.56×–1.47×** 这个区间内,没有一个偏离超过 2 倍,更没有出现"实际上根本
不稀释"或"稀释了两个数量级以上"这种会推翻或极大改写 4.1 节结论的意外
结果。

#### 结论:公式被证实还是证伪?

**函数形式(`∝ 1/L_z`,与 `N`、世界水平尺度无关)被三个独立实验精确
证实,不是近似成立,是数值上逐行吻合**。**精确的几何常数**(`πr/2` 这个
具体系数)**没有被精确证实**——实测数字在 `0.56×–1.47×` 公式值之间浮动,
取决于测的是静态快照还是动态移动、均匀分布还是真实分布,但**没有一次
测量落在"公式错了一个数量级"这种会改写 4.3 节结论的区间**。诚实的说法
是:**4.1 节推导的定性结论(3D 化会把遭遇率按 `1/L_z` 稀释,`N` 救不了
它)被证实;推导给出的具体系数是同类模型里的一个合理估计,实测值在
0.56–1.47 倍之间浮动,不应该被当成精确到个位数百分比的常数来引用**。

#### 回答 4.3 节的问题:`L_z` 该选多少

用实验 3(最贴近真实语义的测量)反推:**比值恢复到 1(不稀释)出现在
`L_z≈6–8`**——恰好和 `attack_range=6.0` 同一个量级,而不是 4.1 节表格
举例的"`L_z=64`,一片高大乔木林"。把 `L_z=64` 代入实测,比值仍然只有
**0.12–0.20**——也就是说,**4.1 节自己举的"合理"例子(60 余单位高的
林冠)其实完全不够浅,实测确认它仍然会把遭遇率打掉 80%–88%**,并不是
一个已经隐含"选浅了就没事"的安全选择。这是本次验证对原推导的一个实质
修正,而不只是复核:**如果目标是"零生态学代价的浅 `L_z`",数字要选在
6–9 个世界单位这个区间,是攻击范围本身的量级,不是森林高度的量级**——
这样的世界严格来说已经不再是"真 3D",而是一个极扁的准 3D,和路线 B 的
物理直觉(离散层)在实践上很接近,只是垂直方向仍然连续。

如果不追求零代价、接受一部分稀释再靠其他手段对冲,`L_z=16`(测出的比值
0.49–0.63)是一个更折中的点。下面把 4.3 节列的第 2、3 条对冲手段用实测
数字重新算一遍:

**对冲 2(放大 `attack_range`)**:要让比值在给定 `L_z` 下恢复到 1,由
4.1 节公式反解 `r_new = 2·L_z/π`:

| 选定的 `L_z` | 需要的 `attack_range` | 相对当前(6.0)的倍数 |
| --- | --- | --- |
| 64(4.1 节例子) | 40.7 | 6.8× |
| 32 | 20.4 | 3.4× |
| 16 | 10.2 | 1.7× |

`L_z=64` 需要的 `40.7` 已经超过 `vision_radius=21.0`——攻击范围比视野
半径还大,这不是"调大一点",是彻底推翻 `config.py` 里"`attack_range=6.0`
故意设得短,给猎物一个体型庇护所"这条已经调好的设计前提(注释原文引用见
2.3 节),连带打开整套已验证的捕食-猎物平衡,**不建议作为 `L_z=64` 的
对冲手段**。`L_z=16` 需要的 `10.2` 只是当前值的 1.7 倍,相对温和,但仍然
需要六种子配对验证才能确认不会打破现有平衡,这里没有测。

**对冲 3(提高密度)**:遭遇率公式里密度线性进入分子,所以要用密度补偿
稀释,需要的密度倍数就是稀释比值的倒数——用实测值(而不是公式值,更
保守):

| 选定的 `L_z` | 需要的密度倍数(1/实测比值) | 对应 `n_max`(以 16384 为基准) | 第 3 节里最接近的实测配置 |
| --- | --- | --- | --- |
| 64 | 6.8×–8.4× | ~111,000–137,000 | 远超 3.2 节测过的最大值(65536,当时已"重争用不可信") |
| 32 | 3.4×–4.1× | ~56,000–67,000 | 超过 32768(测得 ~370 steps/s,642.7 MiB)与 65536 之间 |
| 16 | 1.7×–2.0× | ~28,000–33,000 | 接近 32768 这一档(测得可用,~370 steps/s) |

`L_z=64` 或 `32` 需要的密度倍数,把 `n_max` 推到第 3 节实测范围之外或
边缘(65536 那一档本身"重争用,不可信",不是一个确认稳定的性能点)。
`L_z=16` 配合约 1.7–2.0 倍密度,落在 3.2 节已经实测过、性能可预期的区间
内(32768 那一档),是三个 `L_z` 候选里唯一一个"补偿手段本身不需要冒险"
的组合——但这仍然只补偿了遭遇率的**数量**,不补偿 3.3 节说明的"更大
`n_max`+更大候选列表"这条邻居索引本身的性能代价会跟着倍增,两笔账不能
只算一笔。

**对冲 4(代谢参数)**:如 4.3 节所述,这不是几何问题,本次实验没有
（也不应该在这个实验里）触碰它,需要独立的六种子配对验证。

#### 最终判决

- **`L_z = world_size`(各向同性立方体)**:不值得。3.4 节已经说明这不
  是一个认真的默认假设,本节的实测进一步确认它把遭遇率打到现有值的
  1.4%–2.7%(静态几何测量与动态真实移动测量分别给出这个区间的两端),
  三种独立方法一致,没有任何反转的迹象。
- **`L_z≈64`(4.1 节例子,"一片高大乔木林")**:**不建议**在不重调生态
  参数的前提下直接采用——实测比值只有 0.12–0.20,对冲手段要么需要把
  `attack_range` 推到打破现有设计前提的 40.7(6.8×),要么需要把密度推到
  第 3 节测试范围之外。这不是"不可行",是"需要一次独立于本次调研范围的
  生态学重调"(4.1 节末尾的原话),而且重调的幅度(捕食范围翻近 7 倍,或
  种群翻近 7–8 倍)比 4.2 节讨论的"`carn_cost` 降 33%"量级大得多,值得
  在立项前就说清楚这个数量级差异。
- **`L_z≈16`,配合约 1.7–2.0 倍密度**:是本次实测找出的、性价比最高的
  真 3D 候选点——对冲手段本身没有超出第 3 节已实测的性能/显存范围,但
  它对应的世界形状(垂直 16 个单位,水平 512 个单位,长宽比 32:1)已经
  非常扁,离用户原始设想的"立体世界"有多远需要如实说清楚,不能只报数字
  不报形状。这条路径**没有被证伪,但也没有被这次实验证实为"可行"**——
  它只是证实了"几何账能对上",4.2/4.3 节已经指出的对冲 4(代谢参数)和
  完整的六种子长种子验证一个都没有做,做了才能真正回答"路线 C 值得做"。
- **`L_z≈6–9`(零代价选择)**:几何上完全不稀释,但代价是这样的世界在
  物理直觉上已经不再是"真 3D",这个诚实的权衡必须写进任何后续提案,不能
  藏起来。

## 5. 便宜的替代品

这一节回答"如果不做真 3D,能用多便宜的代价买到多少 3D 的价值"。三个候选,
每个都给出"买到什么/买不到什么"。

### 5.1 地形视线遮挡(line-of-sight occlusion)

`[提案,非结论]`,本次调研**没有实现或测量**,以下是基于现有代码结构的
成本估计,不是实测数字。

**是什么**:现在 `sensors.sense`(`sensors.py:59-64`)只用"观察者自己所在
格子的林冠密度"去统一缩短视野半径(`forest_occlusion`),但**不管观察者
和目标之间隔着什么**——即使中间横着一座山,只要在 `vision_radius` 内就
能看见。真实世界里山会挡视线,这是"3D 地形"里唯一真正影响可见性、而不
只是影响移动/进食的那部分——某种意义上,**这是唯一一条"3D 味道"主要来自
高度而不是来自额外一个自由度"的功能**,而地形的高度场数据本身早就有
(`terrain.height`,不需要新增任何东西)。

**怎么做**:在 `sensors.sense` 判定某个候选者是否可见之前,沿"观察者→
候选者"的连线采样几个中间点(比如 3-5 个等距点,复用 `pos_to_cell` 查
`terrain.height`),如果连线上任意一点的高度显著超过"观察者高度和候选者
高度的线性插值",判定为被遮挡、`closeness` 直接置零。这在计算上是对现有
候选列表 `[n, M]`(`M=9K=216`)**再做几次向量化的插值+查表**,量级上和
现有 `for s in range(R)` 的扇区分箱循环(已经对 `[n, M]` 做 `R=8` 次
`jnp.where`)是同一类操作,不新增数据依赖,不改变任何张量形状。

**成本**:`[推导,非实测]` 大致是在现有 `world_step` 总耗时上增加几个
百分点(几次额外的按元素插值+查表,规模和"扇区分箱"这类已有操作相当,
后者本身只占 `sense`+`predation`+`spatial` 总耗时的一部分)。**这条数字
没有被本次调研实际实现验证**,如实标注——旧报告口头转述的"<5% 步进成本"
是一个合理的量级估计,但既然没有代码可以跑,不应该当作已验证的事实来引用。

**`in_dim` 是否变**:不变。这是一个纯粹的可见性判据修改,不引入新的感知
通道,只是让现有通道(`food`/`prey`/`pred`/`peer`)在"隔着山"的情况下
正确地读到 0 而不是读到一个不该看见的值。**不作废任何种群**。

**买到什么/买不到什么**:买到"山会挡住视线"这个 3D 世界最直观的体验之一
——agent 不会再对着一座山"透视"看到山后面的猎物或水源。**买不到**任何
垂直方向的行为空间——agent 依然不能"爬到高处看得更远"、不能"飞"、不能
在树冠和地面之间做选择,它感知到的世界依然是一张平面,只是这张平面上多了
"有些方向的视线会被地形挡住"这一条规则。

### 5.2 离散 `stratum`

即第 2.2 节的路线 B,这里只重复"买到什么/买不到什么"这个问题,不重复
实现细节。

**买到什么**:一个真正的、agent 可以感知和(在 B2 下)部分控制的垂直
行为维度——"树上更安全但要付出移动代价""树上的资源和地面不同""捕食者
是否也能上树",这些是真 3D 才有的行为分支里最容易先兑现的一批,而且
B1 子方案完全不touch 种群兼容性。

**买不到什么**:连续的垂直位置——不能表达"半山腰""树冠中层"这种介于
离散层之间的状态,朝向依然是平面的一个标量角(一只鸟不能真的"俯冲"),
第 4 节的遭遇率稀释问题在离散分层下**不会显著出现**(因为没有真正的
体积稀释,只是同一个格子被拆成了两个逻辑上独立的子格),这既是优点
(不会像路线 C 那样把捕食者推向灭绝)也是限制(它买不到"3D 空间本身
更稀疏"这个真 3D 最核心的生态学后果——如果调研的初衷之一就是想看"密度
如何随三维空间变化影响演化",离散分层给不出这个答案)。

### 5.3 纯 3D 渲染(路线 A)

即第 2.1 节,这里同样只重复"买到/买不到"。

**买到什么**:视觉上的立体感——山看起来真的是山,相机可以旋转、拉近拉远,
点击拾取可以在三维场景里进行。对着屏幕看的直观体验会有质的提升,而且是
三条路线里唯一"零风险"的一条:不碰仿真、不碰 wire 协议下的数据语义、
不作废种群、显卡绰绰有余。

**买不到什么**:**任何**行为或生态学上的变化——agent 的决策空间、感知
范围、捕食判据、密度分布,一个字节都不会变。这条纯粹是给人看的,不是给
agent 用的。如果用户真正想要的是"看到三维世界里演化出的三维行为"(比如
飞行、掘地、多层次的生态位分化),路线 A 单独不能兑现这一点——它只是让
已经存在的 2D 行为"看起来"发生在一个 3D 场景里。

### 5.4 三者的关系:不是互斥的三选一,是可以叠加的独立选项

值得明确指出:5.1(视线遮挡)、5.2(离散分层)、5.3(纯渲染)这三条
**互相独立,可以同时做**,而且叠加起来能覆盖真 3D 的大部分**视觉**价值
和一部分**行为**价值,同时完全避开路线 C 的两个最大风险(第 4 节的生态
学致命性、以及作废全部已演化种群)。比如"路线 A(相机+高度场网格)+
5.1(视线遮挡)+ 5.2 的 B1 子方案(被动分层影响资源分布)"这个组合,能让
世界看起来是 3D 的、山会挡视线、林冠和地面资源不同,而且**完全不触碰
`in_dim`/`out_dim`/`genome_size`**,是本次调研认为性价比最高的一条路径
——见下一节的决策树。

## 6. 推荐与决策树

### 6.1 明确推荐

**现阶段推荐:路线 A(3D 渲染)+ 5.1(地形视线遮挡)+ 5.2 的 B1 子方案
(被动离散分层,影响资源分布但不新增 agent 可控的行为维度),三者按这个
顺序依次做,每做完一个就足够独立验证和交付,不需要一次性打包。**

理由,直接对应前面几节的发现:

1. 路线 A 是唯一"零风险"的一条——不作废种群、不碰 wire 协议、渲染成本
   在任何现代显卡上都可忽略(第 2.1、3.1 节),而且"世界看起来是立体的"
   这一条本身很可能是用户最先注意到、最直观的诉求。
2. 视线遮挡(5.1)是所有"看起来 3D"的功能里,离"3D 真正改变行为"最近的
   一步——它是唯一一条不需要给 agent 新增感知维度、却能让地形高度真正
   影响决策(而不只是影响移动速度和视野半径这两个标量修正)的机制。
3. 被动分层(B1)在完全不触碰种群兼容性的前提下,让资源分布获得一个
   离散的垂直结构,是介入生态学复杂度成本最低的一步。
4. **路线 C(真 3D)不推荐现阶段做**——第 4 节的推导表明,在任何物理上
   合理的第三维尺度下,遭遇率稀释到现有值的 15-30%,而当前捕食生态是
   一个"降低 `carn_cost` 33% 才勉强不灭绝"的近阈值系统。第 4.3 节末尾
   提议的低成本验证实验**现在已经做完**(见 4.4 节):三个独立方法测出的
   `Z_3D/Z_2D` 比值和推导公式在 0.56–1.47 倍以内吻合,**证实(而非证伪)
   了遭遇率会被稀释这一定性结论**,不是"推导可能有误、实测后发现其实没事"
   ——4.4 节反而发现推导原本举的"`L_z=64`,一片高大乔木林"这个例子并不
   像看起来那么安全(实测比值仍只有 0.12–0.20),真正把稀释压到可忽略
   需要 `L_z≈6–9`,已经不是"深林冠"而是"极扁的准 3D"。在为选定的 `L_z`
   同时规划好对冲手段(4.4 节按实测重新给出了具体数字:`L_z=64` 需要
   `attack_range` 翻 6.8 倍或密度翻 7–8 倍,两者都超出已验证范围;
   `L_z=16` 只需要密度翻 1.7–2 倍,在第 3 节实测范围内)并跑完六种子配对
   验证之前,路线 C 仍然不建议现阶段做——现在缺的不是"知不知道会稀释多少"
   ,而是"生态参数重调"这一步本身还没做。

### 6.2 决策树

```
是否需要三维立体视觉体验(相机可旋转、山看起来是山)?
├─ 是 → 做路线 A。风险为零,随时可以做,和下面的选择无关。
└─ 否 → 跳过本节其余部分。

是否希望地形高度真正影响 agent 的决策(而不只是移动速度/视野半径的标量修正)?
├─ 是,且不想动 in_dim/genome_size → 做 5.1(视线遮挡)。
└─ 否 → 跳过。

是否希望 agent 有一个可感知、部分可控的垂直行为维度(地面/树冠这类离散层)?
├─ 是,且能接受一次种群作废(B2,in_dim 涨约 12%)→ 做 5.2 的 B2 子方案。
├─ 是,但暂时不想作废种群 → 先做 5.2 的 B1 子方案(纯被动分层)。
└─ 否 → 跳过。

是否需要真正连续的三维空间(飞行、掘地、连续高度上的生态位分化)?
├─ 否 → 到这里为止,路线 A + 5.1 + 5.2(B1 或 B2)已经是当前性价比最高的组合。
└─ 是 → 这就是路线 C。第 4.3 节提议的低成本验证实验已经做完(见 4.4 节:
    │  三个独立方法测出的比值落在推导公式的 0.56–1.47 倍以内,证实了
    │  "确实按 1/L_z 稀释"这个定性结论)。在动手重写仿真核心之前:
    │
    ├─ 先按 4.4 节的表格选定 `L_z`(不能悄悄选一个数字,必须写进 docs/
    │  并说明理由——4.4 节已经指出 `L_z=64` 这类"看起来合理"的森林高度
    │  实际仍会打掉 80-88% 的遭遇率),同时按 4.4 节反解出的具体数字
    │  规划补偿手段(`L_z=16` 配约 1.7-2 倍密度是唯一没有超出第 3 节
    │  已验证性能范围的组合;放大 attack_range 到能补偿 `L_z=64` 的
    │  40.7 会打破"短攻击范围=猎物庇护所"这条已调好的设计前提)。
    │
    ├─ 选定的组合几何账能对上(如 `L_z=16`+~1.7-2× 密度)→ 再动手做全量
    │  重写,并预留六种子配对验证(`CLAUDE.md` 的既定纪律)的时间和算力
    │  预算——4.4 节只验证了遭遇率的几何账,没有验证 `carn_cost` 等代谢
    │  参数在新密度/新 `L_z` 下是否仍然成立,这一步仍然是独立的、没有做
    │  过的工作。
    │
    └─ 选定的组合需要把 attack_range 或密度推到超出已验证范围之外(如
       `L_z=64` 或更深)→ 停下来,不要投入全量重写。这不是"技术上不
       可行",是"这套生态参数在这个 `L_z` 下需要一次独立于本次调研范围的
       生态学重新调参",而按 CLAUDE.md 的规矩,生态学重调是一个独立决策,
       不应该和一次架构级重写捆在一起做。
```

## 附录 A:`in_dim`/`genome_size` 精确公式与算例

`Config` 里的公式(`[对应]` `underworld/config.py` 的 `in_dim`/
`brain_params`/`genome_size` 三个 `@property`):

```
in_dim      = 5 * retina_sectors + 3 + 4 * memory_slots + retina_sectors
            = 6 * retina_sectors + 3 + 4 * memory_slots         (含 peer 通道)
brain_params = in_dim * hidden + hidden² + hidden + hidden * out_dim + out_dim
genome_size  = brain_params + trait_dim
```

本次调研用这套公式重新算过的具体数字(`hidden=16` 全程不变,附录里的
"路线 C"一栏假设记忆槽从 4 特征/槽升级到 5 特征/槽、`out_dim` 从 2 升到 3):

| 场景 | `retina_sectors`(或球面分区 `S`) | `memory` 特征/槽 | `out_dim` | `trait_dim` | `in_dim` | `genome_size` |
| --- | --- | --- | --- | --- | --- | --- |
| 历史基线(无 peer 通道、无 size 基因) | 8 | 4 | 2 | 2 | 59 | **1252** |
| 当前 2D | 8 | 4 | 2 | 3 | 67 | **1381** |
| 路线 B2 示例(+1 own-stratum,+8 树冠通道) | 8(平面)+1 层标量+8 树冠通道 | 4 | 3 | 3 | 76 | **1542** |
| 路线 C,`S=12` | 12 | 5 | 3 | 3 | 95 | **1846** |
| 路线 C,`S=24` | 24 | 5 | 3 | 3 | 167 | **2998** |
| 路线 C,`S=48` | 48 | 5 | 3 | 3 | 311 | **5302** |

历史基线一行精确复现了丢失报告口头转述的"1252"这个数字,验证了这份公式和
旧报告在那一部分是一致的。旧报告转述的"2596"在当前公式下找不到一个整数
`S` 精确对应(见第 2.3 节的讨论),推测是一个粗略估计而非精确算出的值;
本附录的路线 C 三行是本次调研独立算出、可以自行代入公式复核的数字,建议
今后引用这份文档里的表格而不是旧报告的单一数字。

## 附录 C:4.4 节验证实验的方法论与脚本

4.4 节的三个实验各对应一个独立脚本,均在会话 scratchpad 完成、**不提交
仓库**,且全程未修改 `underworld/` 任何文件(纯几何/纯读取,`underworld`
只被导入过一次,用来跑一次真实的 `new_world`+`scan_fn` 取真实种群快照):

1. `three_d_get_real_state.py`——用未修改的 `underworld.new_world`+
   `scan_fn` 跑 4000 步(`Config(seed=0)`,默认参数),把 `state.alive`/
   `state.pos`/`state.diet` 存成 `.npz`。这是实验 2、3 用的"真实分布"和
   "真实食性"数据的唯一来源,读到的存活数是 `n_eq=1041`,
   `carnivore_frac=0.126`。
2. `three_d_exp1_geom.py`——2D/3D 环面上的静态遭遇计数(见 4.4 实验 1)。
   `N∈{2000, 16384, 1041}`,`L_z∈{512,384,256,192,128,96,64,48,32,24,16,
   12,8,6}`,每格 24 次重复(`N=16384` 因为单次距离矩阵接近这台显卡当时
   ~5.6 GB 的可用显存,降到 6 次重复,并按行分块(`BLOCK=2048`)计算距离
   矩阵而不是一次性物化整个 `[N,N]` 张量,避免和其他并行 agent 的进程
   互相 OOM)。
3. `three_d_exp2_realpos.py`——读 (1) 存的 `.npz`,固定用真实 `x/y`,`z`
   均匀随机重复 24 次每个 `L_z`,和 (2) 里 `n_eq=1041` 那一档均匀分布的
   比值做对比。
4. `three_d_exp3_predation.py`——固定种群(同一份真实 `alive`/`diet`),
   复刻 `dynamics.predation` 的合格判据和 `argmin`-最近合格猎物的语义,
   用持续随机游走(`turn_sigma=0.5` 弧度/步的朝向高斯扰动,速度
   `0.5×max_speed=4.0`,`dt=0.2` 均取 `config.py` 真实值)跑 300 步,
   `lax.scan` 逐步演化并逐步计数。这一个脚本是三个里唯一同时涉及"移动"
   和"食性过滤"的,也是语义上最接近真实 `world_step` 里 `predation()`
   会看到的量。

全部脚本运行时都设置了 `XLA_PYTHON_CLIENT_PREALLOCATE=false`。运行期间
`nvidia-smi` 显示这台卡仍有约 18.5 GB 被其他进程占用,空闲约 5.6 GB——
这是把 `N=16384` 那一档改成分块计算并减少重复次数的直接原因,也是这几个
脚本的显存峰值应该被读作"这次测量没有撞到显存上限"而不是"精确到 MB"的
原因,与附录 B 记录的争用状况是同一台卡、同一类限制。
